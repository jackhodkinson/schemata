# Identity Column Options Not Reflected in Generated DDL

## Summary

Schemata captures identity columns but discards their sequence options when generating DDL. Columns defined as `GENERATED ... AS IDENTITY (START WITH ... INCREMENT BY ...)` round-trip without the explicit sequence configuration, risking behavioral drift.

## Severity

**Medium** – identity columns remain functional but lose explicit start/increment settings, which can break expectations or lead to data collisions in replicated environments.

## Reproduction

1. Define a column in `schema.sql` with identity options:
   ```sql
   id bigint GENERATED BY DEFAULT AS IDENTITY (START WITH 100 INCREMENT BY 10);
   ```
2. Generate a migration or DDL preview.
3. Output omits everything inside the parentheses.

## Root Cause

- `schema.Column.Identity.SequenceOptions` is populated neither by the parser nor by the catalog loader.
- `internal/planner/ddl.go` never emits sequence options even if they were present.

## Fix Sketch

1. Extend the parser and catalog extraction to collect sequence options into `schema.IdentitySpec.SequenceOptions`.
2. Update differ to compare those options.
3. Teach the DDL generator to render `GENERATED ... AS IDENTITY` clauses with options for both CREATE TABLE and ALTER TABLE flows.

## Notes

PostgreSQL exposes these options via `pg_sequence` and the pg_query AST. Normalizing option order (START, INCREMENT, etc.) will help produce stable diffs.
